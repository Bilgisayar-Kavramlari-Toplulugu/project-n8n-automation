AWS / Terraform / n8n / Caddy

Terraform'u bu proje uzerinden oku.

Bu repo AWS uzerinde n8n + Caddy calistiran bir Terraform kurgusu. Ama burada asil konu sadece "ne deploy ediyor?" degil. Provider ne ise yarar, state neden onemlidir, dosyalar niye bu sekilde bolunur ve bu proje bunu nasil temiz bir operasyon akisina cevirir, onu goreceksin.

Temel mantigi oku Hazir sunum notasina git
AWS

VPC, subnet, route table, security group ve EC2 Terraform ile kuruluyor.

Bootstrap

S3 state, DynamoDB lock ve GitHub OIDC ayri katmanda hazirlaniyor.

Docker

Sunucu ilk acildiginda Caddy ve n8n stack'i otomatik ayaga kalkiyor.

Katman 1: Bootstrap

S3 state bucket, DynamoDB lock, OIDC provider ve GitHub IAM role burada kuruluyor.

Katman 2: Main Terraform

VPC, public subnet, internet gateway, route table, security group ve EC2 burada tanimlaniyor.

Katman 3: user_data

Swap, Docker, Compose, Caddy, n8n ve hardening scriptleri ilk boot aninda calisiyor.

1. Temel mantik

Terraform neyi degistiriyor?

Altyapiyi panelden yonetmek yerine dosyadan tarif ediyorsun. Terraform da mevcut durumla istedigin durum arasindaki farki hesaplayip ne yaratacagini, neyi guncelleyecegini ya da neyi silecegini belirliyor.

Bir cumlede

Terraform, altyapinin kod halidir.

Neden onemli?

Ayni sistemi tekrar tekrar ayni sekilde kurabilirsin.

Asil guc

Degisikligi uygulamadan once plan gosterir, sonra kontrollu uygular.

Terraform olmadan

  • AWS panelinde kaynaklar tek tek acilir.
  • Kim neyi neden acti zamanla belirsizlesir.
  • Tekrar kurulum zorlasir.
  • Ekipte farkli kurulum aliskanliklari dogar.
  • Silme ve guncelleme riski artar.

Terraform ile

  • Altyapi okunabilir dosyalarla tanimlanir.
  • Degisiklikten once plan gorulur.
  • Kaynak iliskileri otomatik cozulur.
  • Versiyon kontrolu ile altyapi tarihi tutulur.
  • Tekrar kurulum ve denetim kolaylasir.
terraform init

Provider ve backend ayarlarini hazirlar.

terraform plan

Ne degisecegini gosterir.

terraform apply

Planlanan degisikligi uygular.

terraform destroy

Terraform'un actigi kaynaklari kontrollu siler.

2. Cekirdek kavramlar

Bir repo okurken bilmen gereken taslar

Bu kavramlar oturdugunda Terraform dosyalari cok daha temiz okunur.

Provider

Terraform'un bir platformla konusma eklentisi. Bu repoda aws var.

Resource

Olusturulacak somut nesne. Ornek: aws_vpc, aws_instance.

Variable

Disaridan ayar vermeni saglar. Region, instance tipi, parola gibi.

Output

Apply bittikten sonra gormen gereken sonucu dondurur.

Data Source

Yeni kaynak yaratmaz, mevcut bilgiyi okur. Bu repoda Ubuntu AMI buradan geliyor.

State

Terraform'un hafizasi. Hangi kaynagi hangi ID ile yonettigini bilir.

Backend

State'in nereye yazilacagini belirler. Bu projede lokal degil, S3 tabanli.

Dependency

Kaynaklarin birbirine bagimliligi. Ornek: subnet, VPC'ye baglidir.

Desired State

Terraform'un omurgasi: kodda ne istiyorsan, onu mevcut durumla karsilastirir.

3. Temel dosyalar

4. Provider mantigi

Provider ne ise yarar?

Provider, Terraform ile bulut saglayicisi arasindaki koprudur.

Terraform Core Yazdigini okur, bagimliliklari cozer, plan cikarir.
→
AWS Provider hashicorp/aws bloklarini AWS API cagrilarina cevirir.
→
AWS API Gercek kaynaklar burada acilir.

Bu projede provider katmani

required_providers ile hashicorp/aws ~> 5.0 seciliyor. Sonra provider "aws" { region = var.region } ile hedef bolge belirleniyor.

required_providers Hangi provider ve surum ailesi kullanilacak?
provider "aws" Provider hangi region uzerinden calisacak?
data "aws_ami" Var olan bilgiyi okur: en guncel Ubuntu 22.04 AMI.

Sunumda tek cumle

"Terraform kendi basina AWS'i tanimaz. AWS provider'i, yazdigim HCL'i AWS'in anlayacagi isteklere cevirir. Yani provider, Terraform icin bir tercumandir."

5. Proje ne kuruyor?

Repoyu iki katmanli oku

Bu repo sadece EC2 acip icine Docker kurmuyor. Once operasyon zeminini, sonra runtime katmanini kuruyor.

Katman 1: Bootstrap

bootstrap/main.tf ana stack'ten once gereken zemini hazirlar.

S3 bucket State dosyasini merkezi ve kalici yerde tutar.
DynamoDB table State locking mekanizmasini saglar.
GitHub OIDC provider Keyless kimlik baglantisi kurar.
IAM role + policy GitHub'un Terraform'u hangi yetkiyle calistiracagini belirler.

Katman 2: Runtime stack

Repo kokundeki main.tf, asil altyapiyi ve boot akisini kurar.

VPC + subnet + route Sunucunun internete cikabilecegi ag omurgasi.
Security group 80 ve 443 acik, SSH ise varsayilan olarak kapali.
EC2 instance Ubuntu 22.04 tabanli runtime.
user_data Swap, Docker, Caddy, n8n ve hardening burada devreye girer.
6. Dosya dosya analiz

Bu repodaki kritik dosyalar ne yapiyor?

Burasi teknik anlatinin en guclu bolumu. Dosya isimleri uzerinden akisi okuyorsun.

main.tf

Projenin runtime omurgasi burada. Provider, S3 backend, ag kaynaklari, security group, key pair, AMI secimi ve EC2 orkestrasyonu ayni akista tutuluyor.

  • backend "s3" ile state'in S3'te duracagi soyleniyor.
  • aws_vpc, aws_subnet, aws_internet_gateway ag zeminini kuruyor.
  • aws_security_group HTTP/HTTPS aciyor, SSH'i CIDR bazli kontrol ediyor.
  • data "aws_ami" "ubuntu" en guncel Ubuntu 22.04 imajini buluyor.
  • aws_instance ilk boot aninda user_data calistiriyor.
  • user_data_replace_on_change = true ile script degisirse instance yeniden alinabiliyor.
resource "aws_instance" "app_server" { ami = data.aws_ami.ubuntu.id instance_type = var.instance_type user_data_replace_on_change = true user_data = join("\n", [...scripts...]) }

variables.tf

Runtime tarafini sabit kalip olmaktan cikarir. Maliyet, guvenlik ve davranis ayarlari burada yasiyor.

  • region hedef AWS bolgesini belirler.
  • instance_type maliyet ve performans dengesini tasir.
  • use_spot_instance ile spot mode acilir.
  • n8n_password sensitive tanimli ve validasyonla korunuyor.
  • ssh_allowed_cidrs ve ssh_public_key birlikte verilirse SSH aciliyor.
  • public_key_path tanimli ama mevcut akista kullanilmiyor.
variable "n8n_password" { type = string sensitive = true validation { condition = length(trimspace(var.n8n_password)) >= 12 error_message = "Guclu parola zorunlu." } }

outputs.tf

Apply bittiginde sonucu okunur hale getirir.

  • public_ip ile dis IP gorunur.
  • n8n_web_ui ile gidilecek adres yazilir.
  • ssh_command ile SSH acik mi kapali mi net gorunur.
  • caddy_status ile reverse proxy durumu ozetlenir.
output "n8n_web_ui" { value = "http://${aws_instance.app_server.public_ip}" }

bootstrap/main.tf

Bu dosya uygulama stack'ini degil, Terraform'un kendi operasyon omurgasini kuruyor.

  • S3 bucket ile remote state kuruluyor.
  • DynamoDB ile state locking aciliyor.
  • GitHub OIDC provider tanimlaniyor.
  • GitHub Actions'in assume edecegi IAM role uretiliyor.
  • Execution policy ekleniyor.
resource "aws_iam_openid_connect_provider" "github" { url = "https://token.actions.githubusercontent.com" client_id_list = ["sts.amazonaws.com"] }

scripts/

Terraform sunucuyu yaratir; bu klasor ise sunucunun icini yasar hale getirir.

  • 00-setup-swap.sh: 2 GB swap aciyor.
  • 01-install-docker.sh: Docker ve Compose plugin kuruyor.
  • 02-deploy-stack.sh: Caddyfile ve docker compose ile n8n stack'ini kaldiriyor.
  • 03-security-hardening.sh: UFW, fail2ban ve SSH hardening yapiyor.
  • Caddyfile su an net olarak :80 uzerinden reverse proxy yapiyor; 443 tarafi domain ve TLS tamamlandiginda anlam kazaniyor.
user_data = join("\n", [ file("scripts/00-setup-swap.sh"), file("scripts/01-install-docker.sh"), templatefile("scripts/02-deploy-stack.sh", {...}), templatefile("scripts/03-security-hardening.sh", {...}) ])

.github/workflows/

Plan, deploy ve destroy hatlari GitHub Actions uzerine kurulmus.

  • terraform-ci.yml: PR acildiginda fmt, validate ve plan calisiyor.
  • terraform-deploy.yml: main branch'ine push olunca apply yapiliyor.
  • terraform-destroy.yml: manuel tetiklenip her seyi kaldirabiliyor.
  • GitHub, AWS'e long-lived key yerine OIDC role ile baglaniyor.
on: push: branches: - main jobs: terraform-apply: ... - run: terraform apply -auto-approve

8. Deploy akis

Bu proje gercekte nasil calisiyor?

Akis su: once zemin kuruluyor, sonra ana stack apply ediliyor, sonra EC2 ilk boot'ta ic scriptleri kosuyor.

Bootstrap calisir

bootstrap/main.tf ile S3, DynamoDB, OIDC ve IAM role kurulur.

Ana proje remote state'e baglanir

backend "s3" sayesinde state lokalden cikar, merkezi hale gelir.

PR acilinca plan alinir

GitHub Actions, OIDC ile AWS'e baglanip terraform plan sonucunu uretir.

Main'e merge olunca apply calisir

terraform-deploy.yml ile terraform apply -auto-approve calisir.

AWS kaynaklari olusur

VPC, subnet, route table, internet gateway, security group, key pair ve EC2 yaratilir.

EC2 ilk boot'ta user_data kosar

Swap, Docker, docker compose stack ve security hardening scriptleri sirayla calisir.

Stack canlanir

caddy dis trafik girisi olur, n8n arka tarafta calisir.

Outputs son resmi verir

Public IP, web arayuzu ve SSH durumu terraform output ile gorulur.

9. Guvenlik ve maliyet

Projede secilen kararlar neden mantikli?

"Ne kuruyor?" sorusu kadar "neden boyle kuruyor?" sorusu da onemli.

SSH varsayilan olarak kapali

Yalnizca ssh_public_key ve ssh_allowed_cidrs birlikte verilirse aciliyor.

OIDC ile keyless erisim

GitHub Actions icinde AWS access key tasimak yerine role assume ediliyor.

Spot instance ile maliyet dusuruluyor

use_spot_instance = true varsayilani maliyeti asagi cekiyor.

No Elastic IP

Ek maliyetten kacmak icin Elastic IP kapali, public IP dinamik.

Caddy front door gorevi goruyor

n8n dogrudan acilmiyor; web trafigi once Caddy'e geliyor. HTTP net, HTTPS ise domain ve TLS tamamlandiginda gucleniyor.

UFW + fail2ban + SSH hardening

Guvenlik sadece AWS security group'a birakilmamis.

Region ve timezone bilincli secilmis

AWS bolgesi eu-central-1, uygulama timezone'u Europe/Istanbul.

Asil deger: operasyon kurgusu

Bu repo sadece sunucu acmiyor; guvenligi, maliyeti ve CI/CD hattini birlikte dusunuyor.

10. Sunum notu

Hazir anlatim metni

Kullanima hazir uc seviye anlatim.

Kisa anlatim

Terraform, altyapiyi kod ile tarif etmeye yarayan bir arac. Bu projede AWS uzerinde n8n ve Caddy calisan bir sistem kuruluyor. Kod tarafinda VPC, subnet, security group ve EC2 tanimlaniyor; sunucu acildiginda scriptler Docker'i ve uygulama stack'ini kuruyor.

Orta detayli anlatim

Terraform'u bir tarif sistemi gibi dusunebiliriz. AWS panelinde tek tek kaynak acmak yerine, kod ile "bir VPC, bir subnet, bir route table, bir security group ve bir EC2 istiyorum" diyoruz. Terraform bu istegi alip AWS provider'i uzerinden gerceklestiriyor.

Bu projede iki katman var. Birincisi bootstrap katmani: state'i S3'e koyuyor, lock icin DynamoDB kullaniyor ve GitHub Actions'in AWS'e OIDC ile guvenli sekilde baglanmasi icin IAM role hazirliyor. Ikincisi asil uygulama katmani: EC2 uzerine Docker kurup Caddy ve n8n'i ayaga kaldiriyor.

Detayli anlatim

Terraform'un gucu, altyapiyi desired state mantigiyla yonetmesinde. Sen sonucu tarif ediyorsun; arac, bu sonuca ulasmak icin eksik kaynagi yaratmak veya mevcut kaynagi guncellemek uzere hareket ediyor.

  • main.tf agi ve sunucuyu kuruyor.
  • variables.tf maliyet, guvenlik ve davranis ayarlarini disaridan aliyor.
  • outputs.tf sonucta hangi IP ve URL ile sisteme gidilecegini veriyor.
  • bootstrap/main.tf remote state ve GitHub OIDC kuruyor.
  • scripts/ klasoru ise instance acildiginda icerde ne olacagini belirliyor.
11. SSS

Muhtemel sorular

Anlatimda en cok gelen sorular burada toplanabilir.

Terraform script dili mi, konfigurasyon dili mi?

Daha cok deklaratif konfigurasyon dili gibi dusunmek gerekir. Adim adim "nasil" degil, sonuc olarak "ne" yazilir.

Provider ile resource arasindaki fark ne?

Provider, hangi platformla konusacagini belirler. Resource ise o platformda yaratacagin somut nesnedir.

State neden kritik?

Terraform bir sonraki degisiklikte neyin kendine ait oldugunu state'ten bilir. O yuzden burada state S3 uzerinde merkezi tutuluyor.

Bu projede neden bootstrap klasoru var?

Cunku ana stack'ten once remote state ve OIDC gibi operasyonel temellerin hazir olmasi gerekiyor.

Docker neden Terraform yerine script ile kuruluyor?

Terraform'un ana gorevi altyapiyi yaratmak. Sunucu icindeki ilk kurulum ise user_data ile yapmak daha dogal.

Neden module kullanilmamis?

Kucuk ve anlatilabilir projelerde modulsuz, dogrudan okunur yapi bazen daha ogretici olur.
